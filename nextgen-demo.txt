NextGen demo feasibility notes - appointment booking / slots / cancel
Updated: 2026-06-16

Short answer
------------
Yes, we can demo a NextGen integration in our app, but true appointment scheduling is not available through the public Patient Access FHIR API. For booking appointments, getting open slots, canceling/rescheduling, etc., we need NextGen Enterprise API sandbox access and the scheduling/appointment routes approved for our app.

Important clarification: this means NextGen API access is required for real booking. If by "Nexon" we mean our own app/company name, Nexon can build the demo UI and mock adapter now, but Nexon cannot actually create/cancel appointments in a NextGen EHR without NextGen Enterprise API credentials + approval.

Why Enterprise API access is required
-------------------------------------
1. NextGen describes Patient Access APIs as read-only patient-data APIs.
   Source: NextGen API overview, https://www.nextgen.com/api
   Relevant text: "Read-Only RESTful API endpoints that provide patient data using the FHIR data standard (DSTU2 & R4)."

2. The same NextGen page describes Enterprise APIs as the read/write integration path for provider-organization apps.
   Source: NextGen API overview, https://www.nextgen.com/api
   Relevant text: "Enterprise APIs" are "JSON-based RESTful APIs (800+ routes)" that "power the category of apps that are used by provider organizations and integrate with NextGen Enterprise."
   The comparison table says Enterprise/FHIR client-facing apps use "RESTful APIs (800+ routes) enabling Read and Write transactions across an expansive array of EHR, PM, and related data using a proprietary NextGen JSON data standard."

3. NextGen's onboarding form explicitly says Appointments are outside USCDIv1, and write access requires answering "No" to the read-only USCDI question.
   Source: NextGen API onboarding questions, https://www.nextgen.com/api-on-boarding/questions
   Relevant text from question 7: "For example, USCDIv1 DOES NOT include Appointments, Financial, or Tasking data." It also says if the app needs any non-USCDI data, or if "WRITE access is required," select "No."
   Relevant text from question 10: "If your app is intended to 'write' information into a NextGen client system, please specify each data element your app would need to create or otherwise alter."

4. NextGen's public Patient Access API page frames the public FHIR API as patient access to personal health data, not operational scheduling.
   Source: NextGen Patient Access API docs, https://www.nextgen.com/patient-access-api
   Relevant text: "The FHIR-based Patient Access API is for developers creating Apps intended to empower patients with access to their own personal health data in a NextGen Enterprise practice via API."
   The same page says for Patient Access / USCDI reads there is no contracting requirement, which is consistent with read-only patient access rather than appointment writes.

5. The public NextGen FHIR R4 capability statement does not advertise scheduling resources.
   Source: NextGen public FHIR R4 metadata/capability statement, https://fhir.nextgen.com/nge/prod/fhir-api-r4/fhir/r4/metadata
   Checked 2026-06-16. Advertised resource types included AllergyIntolerance, CarePlan, CareTeam, Observation, Condition, Coverage, Device, DiagnosticReport, DocumentReference, Encounter, Endpoint, Goal, Immunization, Location, MedicationDispense, MedicationRequest, MedicationStatement, Organization, Patient, Practitioner, PractitionerRole, Procedure, Provenance, Questionnaire, QuestionnaireResponse, RelatedPerson, ServiceRequest, and Specimen.
   It did not advertise Appointment, Slot, or Schedule.

6. Developer Portal registration/source notes say Enterprise API apps must select Enterprise APIs and receive client_id/client_secret for sandbox authentication.
   Source: Intraconnects summary of NextGen portal flow, https://docs.intraconnects.com/docs/ehr-basics/nextgen/pre-requisite/
   Relevant text: "If your app will use Enterprise APIs, select Enterprise APIs." It also says "The issued app credentials (client_id & client_secret) will at that time become active and may then be used for sandbox authentication."

What we need from NextGen
-------------------------
1. NextGen Developer Portal access
   - Portal: https://developer.nextgen.com/
   - Use an organizational email, not a personal email.
   - First person from the org becomes Org Admin.
   - Signup requires email verification and SMS 2FA.

2. App registration in the Developer Portal
   - Org Admin goes to Applications -> Add Application.
   - Enter app name and public description.
   - For appointment scheduling, select Enterprise APIs, not just Patient Access/FHIR.
   - NextGen issues client_id and client_secret.
   - Those credentials become active for sandbox authentication.

3. Onboarding / approval details
   - Form: https://www.nextgen.com/api-on-boarding/questions
   - We must answer that the app is NOT read-only USCDI-only, because appointments are not in USCDI v1 and booking/canceling requires write access.
   - The form explicitly asks what non-USCDI data elements are needed and what write operations the app will create/alter.
   - We should list: appointment availability/open slots, appointment booking/create, appointment cancel/update, appointment reschedule/update, providers/resources, locations, appointment types/reasons.

Which API to use
----------------
Use: NextGen Enterprise APIs.

Do NOT build scheduling against public FHIR Patient Access APIs.
- NextGen's public FHIR R4 capability statement exposes read/search for resources like Patient, Practitioner, Location, Encounter, etc.
- It does NOT advertise Appointment, Slot, or Schedule resources.
- The public FHIR API is mainly for patient access / USCDI clinical data reads, not operational scheduling writes.

Appointment features we need from Enterprise API docs
-----------------------------------------------------
The full Enterprise API docs are behind NextGen's Developer Portal / API Hub, so exact route names/paths need to be confirmed once we have sandbox access. In the docs, search for scheduling / appointments / availability / search ahead. Functionally we need routes for:

1. Get available slots
   - Search available appointment times by provider/resource, location/practice, appointment type/event, date range, duration.
   - NextGen's UI calls this concept Appointment Search Ahead / available time slots.

2. Book appointment
   - Create a scheduled appointment for a patient into a selected slot/resource/location.
   - Inputs likely include patient/person id, provider/resource id, practice/location id, event/appointment type, start time, duration, reason/chief complaint/notes.

3. Cancel appointment
   - Update appointment status or call a cancel endpoint.
   - Need cancellation reason/status handling.

4. Reschedule appointment
   - Usually either update existing appointment time/resource or cancel + create replacement, depending on the Enterprise API route model.

5. Supporting lookup routes
   - Patient search / patient id lookup.
   - Provider/resource lookup.
   - Location/practice lookup.
   - Appointment/event type lookup.
   - Existing appointment lookup by patient/date/provider.

Demo plan
---------
Phase 1: Build mocked NextGen connector now
- Add a NextGen connector interface in our app with methods:
  - listSlots(criteria)
  - bookAppointment(payload)
  - cancelAppointment(appointmentId, reason)
  - rescheduleAppointment(appointmentId, newSlot)
- Back it with mock sandbox-style data so the UI demo works immediately.

Phase 2: Swap in real sandbox adapter
- Once we receive client_id/client_secret and Enterprise API docs, wire OAuth/auth and replace the mock methods with real NextGen Enterprise API calls.
- Keep all create/cancel tests sandbox-only until NextGen confirms test patient + test provider/resource.

Phase 3: Production readiness
- Client/practice authorization in NextGen API Client Portal.
- Scope/route approval for scheduling write operations.
- Audit logging for every booking/cancel/reschedule request.
- Idempotency protection so retrying does not double-book.
- Clear sandbox vs production environment separation.

Risks / blockers
----------------
1. Appointment scheduling is non-USCDI operational data, not public patient-access FHIR data.
2. Exact scheduling endpoints are not public; they require Enterprise API portal/docs access.
3. NextGen onboarding/approval may take time, especially for write access.
4. Need explicit sandbox test data: practice/location, provider/resource, appointment/event types, patient/person ids, and safe test slots.
