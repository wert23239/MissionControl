LiveKit voice agent migration

A practical one-page plan for rebuilding the current hiro/phalanx/voice-agent Vapi-based service on LiveKit while preserving prompts, tools, scheduling behavior, monitoring, and call artifacts.

Executive answer

This is very doable, but it is not a weekend swap. The current service is mostly a Vapi orchestration layer: FastAPI starts calls, builds a Vapi assistant config, receives Vapi webhooks, executes Hiro backend tools, and broadcasts transcript/listen events. LiveKit can replace the Vapi media/orchestration layer, but Hiro would need to own more of the runtime loop: SIP participants, room/job lifecycle, STT/LLM/TTS wiring, interruption/turn detection, transfer/hangup controls, recording, and observability.

The right migration is a parallel LiveKit worker that reuses the existing prompt/tool/business logic where possible and exposes the same backend-facing contract. Keep Vapi as the control while testing LiveKit on a small outbound scheduling flow.

Current architecture snapshot

What exists today

  • src/app.py: FastAPI service, CORS, healthcheck, Datadog-style JSON logging, Auth0 token refresh scheduler.
  • src/routers/call.py: POST /call starts outbound Vapi calls; /call/inbound/{facility_id} answers Vapi inbound assistant requests.
  • src/services/call_handler.py: builds the full Vapi assistant config: prompt, first message, OpenAI model, tool schemas, Deepgram/Soniox/11labs STT config, Cartesia voice, hooks, artifact plan, monitor plan, server webhook secret.
  • src/routers/webhook.py + src/services/webhook_handler.py: handles Vapi tool-calls, transcripts, status updates, hangs, language changes, and end-of-call reports.
  • src/core/state.py: in-memory call sessions + per-call AsyncVoiceFunctions.
  • src/core/connection.py: WebSockets for transcript display and Vapi listen audio forwarding.

Provider stack today

  • Transport / calls: Vapi outbound phone calls with Twilio credentials embedded in the Vapi call payload.
  • LLM: OpenAI gpt-4.1, temperature 0, maxTokens: 200.
  • STT: Deepgram default with Flux General English, keyterms, fallback transcriber, plus client-config overrides for Soniox/others.
  • TTS: Cartesia default voice/model, pronunciation replacements via Vapi chunk formatting.
  • Tools: Python methods on AsyncVoiceFunctions, backed by Hiro backend API functions or mocks.
  • Artifacts: Vapi artifact plan for recordings, logs, structured summary / intent / outcome outputs.

Target LiveKit architecture

1

Backend starts call

Existing Hiro backend/voice API creates a LiveKit room + SIP participant instead of POST /vapi/call.
2

LiveKit worker joins

A Python LiveKit Agents worker receives the room job, loads call context, and starts an AgentSession.
3

Pipeline runs

Deepgram/Soniox STT → OpenAI LLM → Cartesia TTS, with LiveKit VAD/turn detection and interruption handling.
4

Tools execute

LiveKit function tools call the same AsyncVoiceFunctions wrappers and persist results to the backend.
5

Events stream back

Transcript, audio monitor, status, recording, and summaries publish through LiveKit/webhooks into the existing UI/backend.

What ports cleanly vs. what is hard

Easy / reusable
  • Prompt builders in src/prompts/*.
  • Most scheduling, intake, verification, slot, booking, reschedule, and flagging tool logic.
  • Backend request wrappers and mock/real function split.
  • Keyterm generation and client config lookup.
Medium
  • Converting Vapi tool schema generation to LiveKit function tools.
  • Mapping first-message modes, silence hooks, idle prompts, endpointing, and turn timing.
  • Recreating transcript WebSocket payloads expected by the UI.
  • Provider config parity for Deepgram/Soniox/Cartesia/OpenAI.
Hard / risky
  • Inbound call routing parity across facilities and languages.
  • Dynamic transfers and hangup behavior with audited backend events.
  • Recording/log/summary artifacts matching Vapi reports.
  • Production scaling, deployment, secrets, cold starts, and HIPAA/security posture.

Proposed implementation plan

Phase 0 — Spike, no product promises

  • Create a separate livekit-agent worker package beside the existing voice agent service.
  • Start one outbound call through LiveKit SIP to a test number.
  • Hardcode one known facility/patient context and one safe read-only tool.
  • Measure latency, interruptions, transcript quality, and TTS quality against Vapi.

Phase 1 — MVP scheduling call

  • Reuse prepare_call_config pieces, but emit a LiveKit runtime config instead of a Vapi JSON config.
  • Port AsyncVoiceFunctions tools into LiveKit function tools.
  • Publish transcript/tool/status events in the existing Hiro-compatible shape.
  • Persist call IDs with an engine prefix: vapi:{id} vs livekit:{room}/{participant}.

Phase 2 — Parity layer

  • Inbound assistant request equivalent: facility dispatch rule → LiveKit room → backend registration.
  • Listen/monitor UI through room subscription or backend relay.
  • Dynamic transfer with pre-resolve, exact-args validation, audit event, and transfer execution.
  • Recordings, structured summaries, call outcomes, language-change events, and failure cleanup.

Phase 3 — Rollout

  • Add per-facility feature flag: voice_engine = vapi | livekit.
  • Shadow test on staff/test calls, then limited real outbound campaign.
  • Compare booking success, average latency, tool error rate, abandonment, transfer accuracy, and QA flags.
  • Keep Vapi fallback until LiveKit has at least two weeks of stable metrics.

Suggested code shape

hiro/phalanx/voice-agent/src/
  engines/
    base.py                 # VoiceEngine interface: start_call, end_call, transfer, get_monitor_token
    vapi_engine.py           # wrapper around current Vapi behavior
    livekit_engine.py        # room + SIP participant + worker dispatch
  livekit_agent/
    worker.py                # LiveKit Agents entrypoint
    session_factory.py       # STT/LLM/TTS/VAD config from existing agent_config
    tool_adapter.py          # AsyncVoiceFunctions -> LiveKit function tools
    event_adapter.py         # LiveKit events -> current transcript/status payloads
    artifact_adapter.py      # recording + summary persistence
  services/
    call_handler.py          # keep prompt/config helpers; reduce Vapi-specific assumptions
The architectural win is an engine abstraction. Hiro should not let Vapi/LiveKit leak into campaign logic, UI, QA, or backend appointment flows.

Key risks and mitigations

RiskWhy it mattersMitigation
Turn detection regressionsScheduling calls are pause-heavy: DOBs, insurance, doctor names, appointment times.Replay real call audio through Vapi vs LiveKit; tune VAD/end-of-turn thresholds before live rollout.
Tool call behavior changesBooking tools have safety sequencing, e.g. verify slot details before commit.Preserve server-side guardrails in AsyncVoiceFunctions; never rely only on prompt instructions.
Transfer parityBad transfers are operationally visible and risky.Ship transfer last; keep exact-args pre-resolution cache and backend audit events.
Artifact/report mismatchQA, feedback, and analytics may expect Vapi-shaped data.Create an artifact compatibility schema and backfill adapter before using LiveKit for production QA.
Compliance/security assumptionsPatient calls, recordings, logs, and vendor routing are sensitive.Confirm BAA/HIPAA posture, region controls, retention, secrets, recording storage, and access tokens before real PHI.

Go / no-go criteria for an MVP

Quality
  • Transcript WER no worse than Vapi baseline on sampled calls.
  • Median interruption recovery feels natural.
  • No agent speech bleed into user transcript.
Reliability
  • Tool call success/error handling matches Vapi flow.
  • Call cleanup removes sessions and closes room/participant state.
  • Monitoring works for active calls.
Business outcome
  • Can verify patient identity.
  • Can fetch slots, read back details, and book safely.
  • Can produce call summary/outcome for QA.

Bottom line

If the goal is lower vendor lock-in and more control, LiveKit is a strong direction. If the goal is fastest production improvement, keep Vapi and fix specific pain points first. The sweet spot is a side-by-side LiveKit prototype: one outbound scheduling flow, same tools, same prompts, same backend, separate engine. After that, use measured latency/transcript/tool-success data to decide whether to migrate fully.
